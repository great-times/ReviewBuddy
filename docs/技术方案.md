# ReviewBuddy 智能评审助手 · 技术方案

> 版本：v0.1（设计稿）
> 日期：2026-06-13
> 技术栈对齐上层 `Colink` 项目，数据库使用 SQLite。

---

## 1. 产品定位

ReviewBuddy 是一个**智能评审助手**，围绕"评审材料"的全生命周期提供能力，并通过持续沉淀评审经验，让评审材料与模板的质量随使用而提升。

核心功能：

1. **模板管理** —— 评审材料模板的创建、分类、版本化、复用。
2. **AI 生成** —— 基于模板与上下文，由 Agent 生成评审材料。
3. **评审** —— 多角色评审、批注、问题记录、状态流转。
4. **归档** —— 评审材料与评审记录的归档、检索、审计留痕。
5. **自学习闭环（核心壁垒）** —— 评审问题结构化沉淀 → RAG 召回 → 规则提炼 → 模板进化。

## 2. 关于 Hermes 与"自学习"的关键澄清

### 2.1 认知纠偏

LLM / Agent 的权重是冻结的，**不会在使用过程中自动"学会"东西**。用户期望的"随使用沉淀质量"的能力，**不是选某个 Agent 就能白送的，必须在应用层工程化实现**。这恰恰是 ChangeBuddy 的核心价值所在，而非依赖底座模型。

> 结论：Hermes 作为基座**可用**（尤其利于私有化部署），但产品壁垒在于我们自建的"自学习闭环"，而不是模型本身。

### 2.2 适配策略：LLM/Agent 可插拔

由于"开源 Hermes Agent"的具体形态待最终确认，方案采用**适配器模式**（对齐 Colink `internal/service/agent/` 的 `AgentAdapter` 插件机制），把底座抽象掉：

```go
// internal/service/agent/adapter.go
type AgentAdapter interface {
    // 生成评审材料 / 执行评审等推理任务
    Complete(ctx context.Context, req *CompletionRequest) (*CompletionResult, error)
    CompleteStream(ctx context.Context, req *CompletionRequest, onChunk func(Chunk)) error
    // 生成文本向量（用于 RAG）。若 Agent 不支持，可回退到独立 embedding provider
    Embed(ctx context.Context, texts []string) ([][]float32, error)
    Name() string
}
```

候选实现（任选其一接入，互不影响主体）：

| 适配器 | 接入方式 | 适用场景 |
|--------|----------|----------|
| `hermes`（开源 Agent） | HTTP / OpenAI 兼容 API 或 CLI spawn | 私有化、自学习诉求 |
| `openai_compat` | OpenAI 兼容 `/v1/chat/completions` | 兼容兜底（vLLM、Ollama、各类网关） |
| `claude` / `qwen` 等 | 各自 SDK | 备选 |

> 落地建议：先实现 `openai_compat` 适配器把端到端跑通，再把 Hermes 作为一个具体适配器接入——因为多数开源 Agent/推理服务都提供 OpenAI 兼容端点。

## 3. 总体架构（对齐 Colink）

```
Web 前端 (React 18 + Ant Design 5 + Zustand + Vite + TS)
        ↓ REST / WebSocket(流式生成)
后端服务 (Go + Gin)
  - cmd/server/            主入口（路由注册、依赖注入）
  - internal/api/          HTTP 处理器（每资源一个 handler）
  - internal/service/      业务逻辑层
      - template/          模板管理
      - guide/             评审材料生成 + 归档
      - review/            评审流程
      - knowledge/         知识库 / RAG / 自学习闭环 ★
      - agent/             LLM/Agent 适配器（插件化）
  - internal/repo/         数据访问层（原生 SQL）
  - internal/model/        数据模型
  - internal/ws/           WebSocket Hub（流式生成推送）
        ↓
SQLite (modernc.org/sqlite 纯 Go 驱动，无需 CGO)
  + sqlite-vec 扩展（向量检索；或先用应用层余弦相似度）
        ↓ Adapter
Hermes / OpenAI 兼容推理服务
```

分层约束（沿用 Colink）：`api → service → repo`，禁止 `api` 直连 `repo`。

## 4. 技术栈

### 后端
- Go + Gin
- SQLite：`modernc.org/sqlite`（纯 Go，无 CGO）
- 迁移：goose，目录 `sql-change/v{版本}/sqlite/{序号}_xxx.sql`，序号全局递增
- 配置：Viper，`configs/config.yaml.example`（模板，提交）+ `config.yaml`（本地，不提交）
- 向量检索：优先 `sqlite-vec`；若部署受限则应用层计算余弦相似度（数据量小可接受）

### 前端
- React 18 + TypeScript + Vite
- Ant Design 5 + `@ant-design/pro-components`
- 状态：Zustand
- 编辑器：`@monaco-editor/react`（写/编辑评审材料与模板）
- 渲染：`react-markdown` + `remark-gfm` + `rehype-highlight`
- 版本对比：`diff-match-patch`（评审材料/模板版本 Diff）
- 流程图：`mermaid`（评审流程可视化）

## 5. 数据模型（SQLite）

```
templates                 模板主表
  id, name, category, description, content(md), variables(json),
  quality_score, usage_count, current_version, status,
  created_by, created_at, updated_at
template_versions         模板版本历史
  id, template_id, version, content, change_note, created_by, created_at

guides                    评审材料
  id, title, template_id, change_ticket_id, content(md), variables(json),
  status(draft/reviewing/approved/archived), risk_level,
  current_version, created_by, created_at, updated_at
guide_versions            评审材料版本历史
  id, guide_id, version, content, created_by, created_at

change_tickets            评审流程（把评审材料挂到具体评审对象上）
  id, code, title, type, risk_level, planned_at, status,
  rollback_required, owner, created_at
change_steps              评审步骤与风险校验
  id, ticket_id, seq, action, rollback_action, checked, result

reviews                   评审实例
  id, guide_id, guide_version, reviewer, status(pending/approved/rejected),
  decision_note, created_at, finished_at
review_comments           评审批注（行级/段级）
  id, review_id, anchor, severity, category, content, resolved, created_at

review_issues             ★ 评审问题（自学习的原料）
  id, source_review_id, category, trigger_condition, problem_desc,
  correct_practice, change_type, frequency, embedding(blob), created_at

knowledge_rules           ★ 提炼出的审查规则
  id, title, rule_type, pattern, suggestion, derived_from(json ids),
  enabled, hit_count, created_at, updated_at

knowledge_chunks          ★ RAG 知识片段（规则/最佳实践/历史问题统一向量化）
  id, source_type, source_id, text, embedding(blob), created_at

audit_logs                审计留痕
  id, actor, action, target_type, target_id, detail(json), created_at
users / roles             用户与角色（评审会签、权限）
```

> `embedding` 用 BLOB 存 float32 数组；启用 `sqlite-vec` 后改用其虚拟表索引。

## 6. 核心模块设计

### 6.1 模板管理
- 分类树 + 标签；模板含变量占位（如 `{{系统名称}}`），生成时填充。
- 版本化：每次修改产生 `template_versions` 记录，支持 Diff 与回滚。
- 质量分 `quality_score`、使用次数 `usage_count` 由系统度量自动维护。

### 6.2 AI 生成（流式）
1. 用户选模板 + 填变量 + 补充上下文（关联系统、材料类型等）。
2. 后端组装 Prompt：模板内容 + **RAG 召回的同类历史问题与规则**（关键）。
3. 经 `AgentAdapter.CompleteStream` 流式生成，WebSocket 推送前端实时渲染。
4. 生成结果落 `guides`（draft），可继续在 Monaco 中人工编辑。

### 6.3 评审
- 多角色会签：可配置评审人/审批顺序。
- 行级/段级批注（`review_comments`，按 anchor 定位）。
- **AI 预审**：提交评审时先由 Agent 结合 `knowledge_rules` + RAG 自动扫一遍，给出疑似问题清单，人工复核——这是"它好像越来越会审"的体现。
- 评审中确认的问题 → 写入 `review_issues`（结构化），进入自学习闭环。

### 6.4 归档
- approved 后归档，全文检索 + 按评审域/系统/时间维度筛选。
- 审计日志 `audit_logs` 记录全过程操作。

### 6.5 自学习闭环 ★（详见第 7 节）

### 6.6 建议补充的功能板块

| 板块 | 价值 | 优先级 |
|------|------|--------|
| 知识库/经验库 | 自学习引擎底座 | P0 |
| 评审流程/流程编排 | 串起申请→评审→执行→回顾 | P0 |
| 审批流/多人协同评审 | 多角色会签、批注、状态流转 | P1 |
| 评审清单与风险校验 | 自动检查关键风险是否有应对方案（事故高发点） | P1 |
| 评审复盘/事后回顾 | 失败根因反哺知识库（闭环回收口） | P1 |
| 质量度量看板 | 质量分/通过率/模板使用率/问题趋势可量化 | P1 |
| 版本对比 Diff | 模板/评审材料演进可视化 | P2 |
| 权限与审计 | 合规留痕 | P2 |
| 通知集成 | 评审/审批提醒（可参考 Colink 飞书集成） | P2 |

## 7. 自学习闭环详细设计 ★

这是产品核心，**不依赖模型重训**，纯应用层实现：

```
┌─────────────────────────────────────────────────────────────┐
│  1. 沉淀   评审/复盘中确认的问题 → review_issues（结构化）      │
│            字段：类型/触发条件/正确写法/关联材料类型            │
│                          ↓ 向量化                              │
│  2. 入库   写入 knowledge_chunks（规则+最佳实践+历史问题统一） │
│                          ↓                                     │
│  3. 召回   生成/审查时按相似度 + 材料类型召回 Top-K 喂给 Agent │
│            （RAG —— "它好像记住了"的真相）                      │
│                          ↓                                     │
│  4. 提炼   高频问题（frequency 达阈值）半自动提炼为            │
│            knowledge_rules（可复用审查规则）                    │
│                          ↓                                     │
│  5. 进化   规则反哺模板 → 模板版本迭代，quality_score 上升     │
│            质量曲线在度量看板可见                               │
└─────────────────────────────────────────────────────────────┘
                          ↑ 复盘失败根因回流到第 1 步
```

- **召回策略**：`embedding 相似度 × 材料类型匹配 × 规则命中` 加权排序。
- **提炼**：可由 Agent 对同类 `review_issues` 聚类总结成规则草案，人工确认后启用（半自动，保证质量）。
- **可观测**：每条规则记 `hit_count`，模板记 `usage_count`/`quality_score`，让"质量在变好"可量化、可演示。
- **远期可选**：积累足量高质量样本后再做 LoRA 微调，前期完全不需要。

## 13. 开放问题（待确认）

1. **Hermes 真身**：开源仓库地址 / 是否提供 OpenAI 兼容端点 / 是否自带 embedding。确认后即可正式实现 `hermes` 适配器（当前网络检索不可用，未能替你核实）。
2. **embedding 来源**：Hermes 是否提供？否则需指定一个 embedding 服务（如 bge / m3e 本地模型）。
3. **部署形态**：是否需要像 Colink 一样打桌面安装包（Tauri/Electron）？还是 Web 私有化部署即可。
4. **用户体系**：是否对接公司内部 SSO/LDAP，还是内置账号体系。
5. **sqlite-vec 可用性**：部署环境能否加载该扩展；不能则用应用层相似度兜底。
